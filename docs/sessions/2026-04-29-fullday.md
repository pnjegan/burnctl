## Phase 1.5 — Diagnosis

- **Bug 1 (daily unreachable):** root cause `bin/burnctl.js`
  `SUBCOMMANDS` allowlist last updated v4.3, never picked up v4.5.0's
  `daily` command. The shim rejected `daily` before it could reach
  `cli.py`'s dispatch dict.
- **Bug 2 (baseline crash):** `baseline_scanner._scan_mcps()` assumed
  dict-shape configs; list-shape `~/.claude.json` triggered
  `AttributeError`. Caught by `scanner._capture_daily_baseline`'s
  try/except, silently dropping baseline_readings rows.
- **Bug 3 (meta — gate was lying):** `daily_qa.py:score_smoke()`
  whitelisted "unknown command" output as **OK** with the comment
  "pending publish — unknown in published version". No symmetric flip
  back to **DOD** post-publish, so the gate had been silently green
  for 5 days while the flagship `daily` command was unreachable on
  `npm @latest`.

## Phase 1.7 — v4.5.5 hotfix-on-hotfix (1 commit + bump)

- **Root cause:** `db.get_conn()` called `sqlite3.connect()`
  unconditionally, silently auto-creating empty schemaless DBs.
  v4.5.4 unmasked the bug because `cmd_daily` was now reachable but
  doesn't go through `init_db` — it called
  `daily_report.build_daily_brief()` → `db.get_conn()` → empty DB
  created at the npx install dir → 7 subsequent commands' own
  `load_db()` found that empty file (existence check passed) and
  tracebacked.
- **`8ef7f21` fix(db):** `get_conn()` now existence-checks
  `(DB_PATH, ~/.burnctl/data/usage.db)` and returns `None` when
  neither exists. `init_db()` preserves auto-create via shared
  `_open_or_create` helper. `daily_report.build_daily_brief()` adds
  None-guard returning a minimal-shaped 8-key brief; `cmd_daily`'s
  existing printer renders all three score_daily headers via the
  existing `available=False` ELSE branches. **TD-12, TD-13, TD-14**
  filed.
- Released **v4.5.5** to `npm @latest`.
- Post-publish proof: **`18/20 WOW · 2 OK · 0 DOD`** on VPS, clean
  brief on fresh `/tmp` via `npx burnctl@latest daily`, no
  auto-created DBs anywhere.

## Lessons learned

- **Mac smoke test is non-negotiable.** Without it, v4.5.4 (and
  v4.5.0 before it) would have remained silently broken on
  `@latest` indefinitely. The VPS has a real DB at
  `~/projects/burnctl/data/usage.db` that masks every fresh-install
  bug — only running `npx burnctl@latest <cmd>` from a path with no
  DB exposes the truth.
- **The QA gate was lying.** `score_smoke` whitelist + missing TESTS
  entries meant `daily_qa.py` scored `19/20 WOW` while the flagship
  command was unreachable. A gate that passes when the product is
  broken is worse than no gate. Honest gates prevent silent
  regressions; defensive gates (`output.lower()`, header-check in
  `score_daily`) defend against future drift.
- **"Single source of truth" is not abstract — it's the architectural
  cause of bugs.** The shim/cli drift was the cause of v4.5.0's
  `daily` regression (two allowlists in two places, one updated, the
  other not). The `daily_qa` whitelist was the cause of the regression
  going undetected (gate had its own opinion about valid output that
  diverged from reality). Eliminating duplicate sources of truth
  (shim's `SUBCOMMANDS` Set, daily_qa's "unknown command" whitelist)
  removed the entire bug class.
- **Unmasking matters more than fixing.** v4.5.4's COMMIT 1 was
  correct in isolation but exposed v4.5.0's latent `db.get_conn`
  bug. Each unmasked bug needs a dedicated audit before declaring
  the fix complete. The discipline rule "if v4.5.5 reveals another
  cascade bug, we STOP and ship Monday" was the right boundary —
  fortunately the cascade was contained.
- **The npx cache is a hidden actor.** v4.5.4's auto-created empty
  DB at `/root/.npm/_npx/<hash>/node_modules/burnctl/data/usage.db`
  persisted across the v4.5.4→v4.5.5 transition (different cache
  hash for different versions, so v4.5.5 went to a fresh dir and
  worked). Worth remembering for any future hotfix that touches
  caching behavior.
