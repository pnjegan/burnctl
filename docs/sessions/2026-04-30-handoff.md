## What burnctl is (for fresh-session context)

Open-source Claude Code token-waste intelligence tool. github.com/pnjegan/burnctl. Distributed via npm, served via pm2 on tidify-vps (vps3326999), SQLite WAL backing store. Architecture:

- **Mac side** (`tools/mac-sync.py`): polls claude.ai API (browser-bound auth lives in Mac cookies), pushes snapshots to VPS via authenticated webhook every ~5 min
- **VPS side** (server.py + Flask + Jinja templates + SQLite): ingests snapshots, runs scanner, computes insights, serves dashboard at localhost:8080
- **CLI** (cli.py + bin/burnctl.js shim): user-facing commands (daily, scan, accounts, etc.) — distributed via npm

Cross-source visibility (CLI usage from `~/.claude/projects/` JSONL + claude.ai browser sessions) is the product moat. No competitor unifies these.

## Architectural patterns / conventions

- TDs in `TECH_DEBT.md`, format: `### TD-N — title` with **Status / Priority / Files / Context / Fix / Acceptance / Added** fields, file paths in backticks

- Single source of truth principle: cli.py dispatches commands; bin/burnctl.js is a thin passthrough (no SUBCOMMANDS allowlist). Drift between layers is what caused v4.5.0's daily-unreachable bug.
